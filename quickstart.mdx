This guide walks you through using Restack CLI for the first time. You'll learn how to reorganize commits across multiple PRs into clean, stacked branches.

## Before you begin

Make sure you have:

- [Installed Restack CLI](/installation) on your system
- A Git repository with multiple commits you want to reorganize
- Local branches pointing to the last commit of each PR

<Warning>
  Restack creates new branches and modifies your Git history. Make sure you have committed or stashed any working changes before proceeding.
</Warning>

## Understanding the command syntax

The basic Restack command follows this pattern:

```bash
restack <start-commit> <end-commit> [base-branch]
```

### Parameters

<ParamField path="start-commit" type="string" required>
  The commit hash where your reorganization range begins (inclusive)
</ParamField>

<ParamField path="end-commit" type="string" required>
  The commit hash where your reorganization range ends (inclusive)
</ParamField>

<ParamField path="base-branch" type="string" default="origin/feature/anytime">
  The branch to use as the base for the first PR. Can be a local or remote branch.
</ParamField>

## Your first restack

Let's walk through a complete example:

<Steps>
  <Step title="Identify your commit range">
    First, find the commit hashes for your range. Use `git log` to view your commits:
    
    ```bash
    git log --oneline
    ```
    
    Look for the first and last commits you want to reorganize. For example:
    - Start commit: `938fa82e`
    - End commit: `f80ce55b`
  </Step>
  
  <Step title="Ensure branches mark PR boundaries">
    Restack identifies PR boundaries by finding commits with local branches. Make sure each PR's last commit has a branch pointing to it.
    
    Check which commits have branches:
    ```bash
    git log --oneline --decorate
    ```
    
    Create branches if needed:
    ```bash
    git branch feature/my-pr <commit-hash>
    ```
  </Step>
  
  <Step title="Run the restack command">
    Execute Restack with your commit range and base branch:
    
    ```bash
    restack 938fa82e f80ce55b origin/feature/anytime
    ```
    
    Restack will:
    1. Analyze the commit range
    2. Identify PR boundaries based on branches
    3. Display the discovered PRs and reorganization plan
  </Step>
  
  <Step title="Review and confirm the plan">
    Restack shows you the PR ranges it discovered and the branch chain structure:
    
    ```
    📚 Discovered PR ranges:
      feature/add-api ← 938fa82 ~ a3b4c5d (3 commits)
      feature/add-tests ← d6e7f8g ~ f80ce55 (2 commits)
    
    📚 Re-stack plan:
    Base: origin/feature/anytime
      └─ feature/add-api
          └─ feature/add-tests
    
    ⚠️  Proceed with this plan? (y/n)
    ```
    
    Review the plan carefully and type `y` to continue.
  </Step>
  
  <Step title="Handle conflicts (if any)">
    If cherry-pick conflicts occur, Restack pauses and offers options:
    
    ```
    ⚠️  Conflict detected. How do you want to proceed?
       1) Resolve conflicts and continue (git cherry-pick --continue)
       2) Skip this commit (git cherry-pick --skip)
       3) Abort entire operation (git cherry-pick --abort)
    Choose (1/2/3):
    ```
    
    To resolve conflicts:
    1. Select option `1`
    2. Fix the conflicts in your editor
    3. Stage the resolved files: `git add .`
    4. Press Enter to continue
  </Step>
  
  <Step title="Address type checking (if enabled)">
    After each PR is reorganized, Restack runs TypeScript type checking. If errors are found:
    
    ```
    🔍 Type checking... (feature/add-api)
       ❌ Type checking failed!
    
    ⚠️  Type errors detected. How do you want to proceed?
       1) Fix type errors and continue
       2) Abort entire operation
    Choose (1/2):
    ```
    
    Select option `1`, fix the errors, and press Enter to retry type checking.
  </Step>
  
  <Step title="Push the new branches">
    Once reorganization completes, Restack asks if you want to push:
    
    ```
    💡 Push all created branches to remote? (y/n)
    ```
    
    Type `y` to push all branches with `--force-with-lease`, or `n` to push manually later.
  </Step>
</Steps>

## What happens during reorganization

Here's what Restack does behind the scenes:

### Temporary branches

Restack creates temporary branches with the `RESTACK-` prefix for each PR. For example:
- `RESTACK-feature/add-api`
- `RESTACK-feature/add-tests`

These temporary branches are renamed to the final branch names only after type checking passes.

### Cherry-pick sequence

For each PR:
1. Create a new temporary branch from the previous PR's branch (or base branch for the first PR)
2. Cherry-pick all commits in the PR range sequentially
3. Run type checking on the temporary branch
4. Rename the temporary branch to the final name

### Branch chain structure

The result is a clean branch chain where:
- First PR builds on the base branch
- Second PR builds on the first PR's branch
- Third PR builds on the second PR's branch
- And so on...

## Example workflow

Here's a complete real-world example:

```bash
# View your commit history
git log --oneline --decorate

# Run restack with your commit range
restack 938fa82e f80ce55b origin/feature/anytime

# Review the plan and confirm with 'y'

# If conflicts occur, resolve them:
# 1. Edit the conflicted files
# 2. git add .
# 3. Press Enter

# If type errors occur, fix them:
# 1. Edit the files with errors
# 2. git add .
# 3. git commit --amend --no-edit (if prompted)
# 4. Press Enter

# Push the new branches when prompted with 'y'
```

## Tips for success

<AccordionGroup>
  <Accordion title="Always commit your work first">
    Restack modifies branches and commit history. Save your work before running it:
    ```bash
    git add .
    git commit -m "Save work before restack"
    ```
  </Accordion>
  
  <Accordion title="Use descriptive branch names">
    Since Restack uses branch names to identify PRs, choose clear, descriptive names:
    - Good: `feature/add-user-auth`, `fix/payment-bug`
    - Avoid: `temp`, `branch1`, `test`
  </Accordion>
  
  <Accordion title="Keep commit ranges manageable">
    Start with smaller ranges (5-10 commits) to get familiar with the tool before tackling larger reorganizations.
  </Accordion>
  
  <Accordion title="Review the plan carefully">
    Always read the reorganization plan before confirming. Make sure the PR boundaries and branch chain match your expectations.
  </Accordion>
</AccordionGroup>
